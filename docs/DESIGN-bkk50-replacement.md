# bkk50 Replacement — Linux LXC NAT Router

**Status:** DRAFT
**Author:** Network Team
**Date:** 2026-06-04
**Version:** 0.1

---

## 1. Executive Summary

Replace the bkk50 MikroTik CCR2004 (RouterOS) with a **Linux LXC container
on bkk07** running `nftables` + `bird`. The container takes over bkk50's
role as the internal stateful NAT router and the announcement origin for
`160.22.181.181/32`.

**Edge stays stateless.** bkk00 / bkk20 (RouterOS eBGP edges) are *not*
modified — they continue to announce the rotko prefix to upstreams
without any conntrack on their hot path.

**Motivation:**

- All config (`nftables`, `bird`) is generated by the in-repo
  `deploy/*/gencfg.sh` toolchain — the same tooling already used for
  bkk06/07/08. Eliminates the RouterOS CLI as a unique configuration
  surface.
- LXC is portable: bkk07 fails → container migrates to bkk08 in seconds.
  bkk50 hardware failure today requires a manual rebuild.
- CI deploy of router config becomes possible via existing
  `deploy-nftables.yml` / `deploy-bird.yml` workflows.
- Decouples the network-layer NAT role from a single physical box that
  isn't part of the regular Proxmox/Ansible flow.

**Non-goals:**

- Not moving NAT/conntrack onto bkk00 or bkk20. Those stay stateless.
- Not changing the anycast announcements from bkk06/07/08 host BIRD.
- Not redesigning the SAX traffic path on initial cutover (see §6).

## 2. Current Architecture (bkk50)

```
                Internet
                   │
            ┌──────┴───────┐
            ▼              ▼
         bkk00          bkk20         eBGP edge — STATELESS
         (RouterOS)     (RouterOS)
            │              │
            └──────┬───────┘  iBGP via RR VLAN 100 / qnq-400-100
                   ▼
                 bkk50               internal NAT router — STATEFUL
                 (RouterOS CCR2004)  loopback 160.22.181.181/32
                   │
            ┌──────┼──────┐
            ▼      ▼      ▼
         bkk06   bkk07  bkk08      Proxmox hosts + container HAProxy
```

bkk50 currently provides:

| Function | Detail |
|---|---|
| `160.22.181.181` announcement | `/32` loopback advertised via iBGP to bkk00/20 |
| External-to-internal DNAT | 22786/87/88 → host SSH; 178XX → IPMI; haproxy CT ports |
| Source NAT for outbound | mgmt subnets snat to `160.22.181.181` |
| Connection tracking | All forwarded flows stateful |
| WireGuard endpoints | `wg_rotko` (mesh) + `bkk_sax_wg` (Saxemberg client) |

## 3. Target Architecture

```
                Internet
                   │
            ┌──────┴───────┐
            ▼              ▼
         bkk00          bkk20         eBGP edge — STATELESS (unchanged)
         (RouterOS)     (RouterOS)
            │              │
            └──────┬───────┘  iBGP via RR VLAN 100 / qnq-400-100
                   ▼
            LXC: bkk-rt-01      Linux + nftables + bird, runs on bkk07
            (160.22.181.181)    HOST: bkk07 vmbr0 (mgmt) + vmbr2 (public)
                   │
            ┌──────┼──────┐
            ▼      ▼      ▼
         bkk06   bkk07  bkk08
```

The LXC is just another iBGP RR client to bkk00/20, identical in role to
the bkk06/07/08 host BIRD instances — only the prefixes it advertises and
the NAT rules it carries differ.

## 4. New Site in `network.json`

Proposed `bkk-rt-01` entry:

```json
"bkk-rt-01": {
  "router_id": "10.155.255.50",
  "bgp_rr_v4": "10.155.100.50",
  "bgp_rr_v6": "fd00:155:100::50",
  "public_v4": "160.22.181.181",
  "public_v6": "2401:a860:181::181",
  "internal_v4": "10.50.0.0/24",
  "internal_v6": "2401:a860:1050::/64",
  "management": "192.168.50.1/24",
  "role": "internal_nat_router",
  "host": "bkk07",
  "physical_interfaces": {
    "management": "eth0",
    "public": "eth1"
  }
}
```

- `router_id` reuses `10.155.255.50` (was bkk50's logical slot).
- `bgp_rr_v4 / _v6 = 10.155.100.50 / fd00:155:100::50` — new RR client
  loopbacks; bkk00 and bkk20 RR filters need an `accept` rule for these.
- `public_v4 = 160.22.181.181` — same IP, now lives on the LXC's loopback.
- `host = "bkk07"` — Proxmox host. Add an HA pair on bkk08 later (§7).

`generated/bkk-rt-01-bird.conf` and a corresponding nftables file would
be produced by the existing `deploy/bird/gencfg.sh` / `deploy/nftables/gencfg.sh`
once the entry is in place. Both generators already handle Linux targets;
no template changes expected, only the `role`-specific output sections
may need a small new branch (see §5).

## 5. Generator changes required

### `deploy/bird/gencfg.sh`

- New role branch for `internal_nat_router`: announce `public_v4/32` and
  `public_v6/128` to iBGP only; do NOT export to eBGP filters.
- Standard RR client iBGP sessions to `10.155.100.1` (bkk00) and
  `10.155.100.2` (bkk20).

### `deploy/nftables/gencfg.sh`

- Translate the bkk50 dstnat rules (currently in
  `backup/mikrotik/bkk50.rsc:910-1000+`) into nftables syntax:

  ```nftables
  # dstnat: external port → internal host
  ip daddr 160.22.181.181 tcp dport 22786 dnat to 192.168.76.1:22
  ip daddr 160.22.181.181 tcp dport 22787 dnat to 192.168.77.1:22
  ip daddr 160.22.181.181 tcp dport 22788 dnat to 192.168.78.218:22
  # ... haproxy CT forwards, IPMI, 207XX, etc.
  ```

- snat for outbound:

  ```nftables
  ip saddr 192.168.0.0/16 oifname WAN snat to 160.22.181.181
  ```

The rule set is mechanical; the existing `bkk50.rsc` is the
specification. Each `dstnat` and `srcnat` line maps 1:1 to nftables.

## 6. SAX Traffic

bkk50 today terminates two WireGuard interfaces:

- `wg_rotko` (172.31.0.0/16) — internal mesh, every site member
- `bkk_sax_wg` — Saxemberg client peering

Both can be moved onto the new LXC (`wireguard-tools` + standard `wg`
interfaces). The keys and peer configs are in the bkk50 backup;
re-import after IP migration.

Open question: does SAX need any port forwards specific to its
infrastructure (192.168.111.x / 192.168.112.x) that aren't already
covered by the generic NAT rules? Audit `bkk50.rsc` lines 900–1500 for
SAX-specific comments before cutover.

## 7. HA Pair (Phase 2)

Initial deployment is one LXC on bkk07. To eliminate the bkk07-host
single-point-of-failure:

- Provision `bkk-rt-02` LXC on bkk08
- VRRPv3 / keepalived between them for `160.22.181.181`
- conntrackd between for stateful failover
- BIRD on both announces `/32` with weighting so only the master is
  active under normal conditions

Defer until §1 is stable in production.

## 8. Migration Phases

Each phase is independently rollback-able.

### Phase A — Provision LXC (no cutover)

1. `pct create` Debian 12 LXC on bkk07 with public + mgmt bridges
2. Install nftables, bird, wireguard-tools
3. Add `bkk-rt-01` entry to `network.json`
4. Run `deploy/bird/gencfg.sh bkk-rt-01` and `deploy/nftables/gencfg.sh bkk-rt-01`
5. Deploy generated configs to the LXC (manual first time)
6. LXC has `160.22.181.181` on `lo` but BIRD does NOT yet announce it
7. Validate: nftables loads, bird daemon up, RR sessions to bkk00/20 established

### Phase B — Add RR client to bkk00 / bkk20

1. Update RR-CLIENT-IN-v4 / -v6 filter on bkk00 to accept `10.155.100.50`
2. Same on bkk20
3. Establish iBGP from LXC; verify session up, advertisements empty

### Phase C — Parallel run

1. LXC BIRD enables `160.22.181.181/32` announcement
2. Both bkk50 AND LXC advertise the same prefix; iBGP MED/AS-path
   tie-break picks one. Verify which one bkk00 prefers via
   `/ip route print where dst-address=160.22.181.181/32 detail`
3. Adjust LXC's local-preference upward so LXC wins
4. Verify external connectivity to `160.22.181.181:22786` still reaches
   bkk06 SSH (traffic now flows through LXC)
5. Soak for 24h, watch `iperf3` throughput and nftables connection table

### Phase D — Cut over

1. Stop bkk50 BIRD / withdraw its `160.22.181.181/32` announcement
2. All traffic now flows through LXC
3. Update CI workflow secrets if SSH host fingerprint changed
   (host_keys may differ; pre-deploy `~/.ssh/known_hosts` update)

### Phase E — Decommission bkk50

1. Power off bkk50 MikroTik
2. Update `docs/` and `network.json` to remove or archive bkk50
   references
3. Optionally retain bkk50 as cold-standby hardware

## 9. Risks

| Risk | Mitigation |
|---|---|
| nftables throughput lower than MikroTik hw-accel | Benchmark with iperf3 before cutover; LXC scale is well within Linux NAT capability |
| bkk07 host reboot drops LXC mid-cutover | Run §7 HA pair before declaring done; bkk07 reboots are rare and planned |
| BGP best-path tie during parallel run | Explicit local-preference bump on LXC's announcement; documented in §8 Phase C |
| WireGuard endpoint IP change breaks SAX/wg_rotko peers | Coordinate key + endpoint migration with peers; window during off-hours |
| Host SSH key change confuses CI's known_hosts | Pre-populate target host fingerprint in workflow secret |

---

## 10. Rollback

Any phase: re-enable bkk50 BIRD announcement; LXC BIRD withdraws.
Traffic shifts back to bkk50 within BGP convergence (<30s). NAT state
on bkk50 is preserved during parallel run because conntrackd has not
been configured between bkk50 (MikroTik) and LXC; in-flight connections
through LXC will reset on rollback. Plan rollback for low-traffic
windows.
